Contexto

El problema en locales gastronómicos

Bares, cafeterías y restaurantes pequeños siguen operando con toma de pedidos manual, generando fricciones en toda la cadena de atención.

Demoras en la atención: El cliente espera a que alguien llegue a la mesa para iniciar el pedido, incluso en momentos de poca carga.
Dependencia del personal: En locales chicos, la misma persona atiende, toma pedidos y prepara, creando cuellos de botella constantes.
Errores en la toma: Pedidos anotados mal, incompletos o comunicados de forma deficiente a cocina generan re-trabajo y mala experiencia.
Sin trazabilidad: El negocio no sabe el estado de cada pedido ni tiene datos de tiempos, errores o productos más solicitados.
Sin información para decidir: Sin datos concretos, es imposible medir tiempos, rendimiento o satisfacción del cliente.

Propuesta

Mesa CLICK

Una app web PWA que digitaliza la toma de pedidos en mesa sin reemplazar al personal, sino dándole mejores herramientas para operar.

Núcleo del producto
QR por mesa
Cada mesa tiene un QR único. El cliente lo escanea y accede a la carta digital desde su celular, sin instalar nada.
Para el negocio
Dashboard de gestión
El personal ve y gestiona pedidos en tiempo real. Confirma, actualiza estados y sigue la operación sin papel ni intermediarios.
Beneficio — cliente
Pide cuando quiere, sin esperar al mozo
Beneficio — personal
Menos interrupciones, operación más clara
Beneficio — negocio
Datos reales y menor tiempo de atención

Usuarios / Personas

¿Quiénes lo usan?

Cuatro actores con necesidades distintas. El cliente es el único que no necesita cuenta ni login.

AD
Admin principal
Dueño o responsable del negocio. Configura el sistema completo: sucursales, mesas, carta y usuarios internos.
Multi-sucursalConfiguración global
EN
Encargado de sucursal
Gestiona mesas, carta y tiempos de demora estimados de su sucursal específica.
Sucursal propiaConfigura carta
RE
Recepcionista / Mozo
Recibe, confirma y actualiza el estado de los pedidos desde el dashboard. Es el operador del día a día.
Dashboard liveGestión de estados
CL
Cliente / Comensal
Escanea el QR, ve la carta, realiza el pedido y sigue su estado desde el celular. No necesita cuenta.
Sin loginSolo QR

Flujo mínimo

Happy path

El recorrido completo de una experiencia exitosa, desde la configuración inicial hasta el cierre del pedido.

1
Admin configura el sistema
Crea sucursal, carga mesas y arma la carta digital con productos y variantes.
2
Cliente escanea el QR
Desde la mesa abre la carta en su celular, selecciona productos y confirma el pedido.
3
Recepcionista confirma
Ve el pedido en el dashboard en tiempo real y lo pasa a "Preparando".
4
Cliente sigue el progreso
Ve el estado actualizado y la demora estimada desde su vista de mesa.
5
Pedido listo
Estado pasa a "Listo". El cliente puede agregar más ítems o pedir la cuenta.
6
Cierre del pedido
El recepcionista cierra el pedido. Queda registrado para los reportes del negocio.

Planificación

Roadmap — 8 sprints

Cada sprint representa ~2 semanas de trabajo. El Sprint 0 es la base de todo: decisión de stack, arquitectura y organización del equipo.

Sprint 0
Definición de stack + arquitectura + setup del entorno
Sprint 1a
Configuración de negocio (tenant, sucursales)
Sprint 1b
Auth + usuarios internos (magic link email)
Sprint 2
Carta digital + mesas + generación de QR
Sprint 3
Recepción de pedidos + estados en tiempo real
Sprint 4
Acciones del cliente (agregar ítems, pedir cuenta)
Sprint 5
Reportes básicos (tiempos, productos populares)
Sprint 6
QA, polish visual, deploy a producción
Sprint 0 no es setup mecánico. Es donde el equipo decide framework, base de datos, estrategia de auth, estructura del monorepo y cómo se organiza el trabajo entre los 3 desarrolladores.

Backlog

Épicas y user stories — MVP

Hacé clic en cada épica para ver las user stories. Roles asignados: Back, Front e Int (Integración) para los 3 devs del equipo.

Sprint 0 Definición de stack y setup del entorno ▶
TodosDecisión de stack: elegir framework back, front, base de datos y estrategia de auth en base a criterios del equipo
BackInicializar monorepo con estructura de carpetas, dependencias base y scripts compartidos
BackConfigurar entorno de desarrollo: Docker / entorno local, variables de entorno, base de datos inicial
FrontSetup del proyecto frontend: bundler, estilos base, estructura de rutas y primera pantalla funcional
IntDefinir convenciones: estructura de APIs, nombrado de branches, flujo de PRs y board de trabajo del equipo
Sprint 1a Configuración de negocio (tenant + sucursales) ▶
BackEndpoint para crear un tenant (negocio): nombre, slug y configuración base
BackModelo de base de datos: Tenant + Sucursal (id, nombre, dirección, demora estimada)
FrontFormulario de onboarding — creación del negocio con validación inline
IntFlujo completo: obtener tenant logueado + agregar primera sucursal post-registro
Sprint 1b Auth con magic link (email) ▶
BackEndpoint de solicitud de magic link: recibe email, genera token único y dispara el envío del correo
BackEndpoint de verificación: valida token, crea sesión y redirige al panel del negocio
FrontPágina de login: input de email, estado "revisá tu correo" y redirect post-verificación
IntInvitación de usuarios internos: el admin genera un magic link para invitar a un encargado o mozo
Sprint 2 Carta digital + mesas + QR ▶
BackCRUD de productos: nombre, descripción, precio, variantes y adicionales por sucursal
BackCRUD de mesas: por sucursal, con capacidad, número y estado activo/inactivo
BackEndpoint que genera los datos del QR con la URL pública de acceso a la mesa
FrontPanel admin: ABM de carta con secciones y categorías de productos
FrontVista pública del cliente (sin login): carta digital con foto, precio y variantes
IntGeneración y descarga del QR por mesa desde el panel de administración
Sprint 3 Pedidos + estados en tiempo real ▶
BackEndpoint para crear un pedido: mesa, lista de ítems y notas opcionales
BackEndpoint para actualizar estado: Recibido → Preparando → Listo → Cerrado
BackEndpoint de pedidos activos por sucursal para el dashboard del recepcionista
FrontDashboard recepcionista: tabla de pedidos con acciones de cambio de estado
FrontVista cliente: confirmación de pedido + seguimiento de estado con demora estimada
IntTiempo real: al cambiar el estado, el cliente y el recepcionista se actualizan sin recargar la página
Sprint 4 Acciones del cliente post-pedido ▶
BackEndpoint para solicitar la cuenta: marca el pedido y notifica al recepcionista
BackEndpoint para agregar ítems adicionales a un pedido existente
FrontBotones "Pedir cuenta" y "Agregar más" en la vista del cliente con confirmación visual
IntNotificación en tiempo real al dashboard cuando el cliente solicita la cuenta
Sprint 5 Reportes básicos ▶
BackEndpoint de métricas: pedidos por día, tiempo promedio de atención por sucursal
BackEndpoint de productos más pedidos con ranking por sucursal y rango de fechas
FrontPanel de reportes: gráficos de pedidos diarios y tabla de productos populares
IntFiltros de fecha y sucursal en los reportes desde el panel del admin
